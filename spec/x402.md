## Abstract

x402 is a stateless settlement-gated HTTP protocol that allows
servers to require verifiable economic settlement as a precondition
for executing a request.

This document defines the canonical protocol specification.

### Specification Stability

Version 1.0 of this specification is frozen.

Future revisions of x402 will be published as new specification
versions.

Implementations conforming to Version 1.0 MUST follow the wire
semantics defined in this document.

## 1. Introduction

HTTP status code **402 Payment Required** has existed since the early
days of the web but was never formally standardized.

x402 defines a deterministic protocol for using HTTP 402 to require
verifiable economic settlement before executing a request.

In x402, payment functions as an authorization primitive.

Instead of relying on accounts, API keys, or subscriptions,
the client proves payment directly as part of the request flow.

x402 therefore converts economic settlement into a stateless
authorization mechanism for HTTP resources.

The protocol invariant is that request execution is conditioned
only on verifiable settlement proof and deterministic request binding.

## 2. Conventions and Terminology

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
"SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY",
and "OPTIONAL" in this document are to be interpreted as described
in RFC 2119 and RFC 8174.

This specification uses the following encoding conventions:

- **base64url**: Base64 URL-safe encoding without padding.
- **`<hex>`**: Lowercase hexadecimal encoding of bytes.
- **`<sha256>`**: A 32-byte SHA-256 digest encoded as `<hex>`.

Terminology:

- **Challenge**: The JSON object carried in the `X402-Challenge` header.
- **Proof**: The JSON object carried in the `X402-Proof` header.
- **Nonce UTXO**: The specific UTXO referenced by `nonce_utxo` in the challenge.
- **Settlement Layer**: The transaction validation and ordering system
  used to verify the payment transaction referenced by the proof.

### Base64 Encodings

Two base64 encodings are used in this specification.

• `base64url` — URL-safe base64 encoding without padding.  
  Used for HTTP headers (`X402-Challenge`, `X402-Proof`).  

• `base64` — standard base64 encoding.  
  Used for binary payloads such as `rawtx_b64`.  

### Canonical JSON Encoding

All JSON objects defined in this specification MUST be serialized
deterministically before hashing.

Canonical serialization MUST follow the JSON Canonicalization Scheme
defined in RFC 8785.

The following rules therefore apply:

• Object keys MUST be sorted lexicographically.  
• Whitespace outside strings MUST NOT be included.  
• Numbers MUST be encoded in their shortest decimal form.  
• Strings MUST use UTF-8 encoding.  

Implementations that cannot implement RFC 8785 MUST use a deterministic
equivalent that guarantees identical byte serialization across
independent implementations.

## 3. Protocol Overview

The x402 protocol follows a simple request-response pattern.

1. The client requests a protected resource.
2. The server responds with HTTP 402 and a challenge.
3. The client performs settlement.
4. The client retries the request with a payment proof.
5. The server verifies the proof and, if valid, serves the resource.

```
Client → Request resource
Server → 402 Payment Required (challenge)
Client → Retry request with proof
Server → 200 OK
```

### Sequence Diagram

```
Client                     Server
  |                         |
  |  Request Resource       |
  |------------------------>|
  |                         |
  |  402 Payment Required   |
  |  + Challenge            |
  |<------------------------|
  |                         |
  |  Request + Proof        |
  |------------------------>|
  |                         |
  |  Resource Response      |
  |<------------------------|
```

The server MUST respond with HTTP 402 when payment is required and a valid
proof is not present on the request.

The client MUST include a valid `X402-Proof` header when retrying a request
with settlement proof.

The server MUST treat authorization as derived solely from verification of
the proof and MUST NOT require client accounts, API keys, or subscriptions
as a condition of x402 authorization.

## 4. HTTP Challenge

### X402-Challenge

The `X402-Challenge` header MUST contain a base64url encoded JSON object
defining the settlement conditions for the request.

Example:

```
HTTP/1.1 402 Payment Required
X402-Challenge: <base64url(JSON)>
```

When payment is required, the server MUST respond with HTTP 402 and MUST
include exactly one `X402-Challenge` header.

The server SHOULD include an `expires_at` value that limits the validity
window of a challenge.

### Challenge Object

The challenge object MUST contain the following fields.

```
{
  "v": 1,
  "scheme": "bsv-tx-v1",

  "domain": "api.example.com",

  "method": "GET",
  "path": "/v1/resource",
  "query": "",

  "req_headers_sha256": "<hex>",
  "req_body_sha256": "<hex>",

  "amount_sats": 50,

  "payee_locking_script_hex": "<hex>",

  "nonce_utxo": {
    "txid": "<hex>",
    "vout": 0,
    "satoshis": 1,
    "locking_script_hex": "<hex>"
  },

  "expires_at": 1700000000,

  "require_mempool_accept": true
}
```

Rules:

• `v` MUST indicate the protocol version  
• `scheme` MUST define the settlement method  
• `amount_sats` MUST define the minimum payment required  
• `expires_at` MUST be a UNIX timestamp  

The server MUST reject challenges whose `v` value it does not support.

Additional requirements:

- **`domain`**: MUST match the authority component of the HTTP request
  target (for example the `Host` header in HTTP/1.1 or the `:authority`
  pseudo-header in HTTP/2 and HTTP/3).
- **`method`**: MUST be the HTTP method to which the challenge binds.
- **`path`**: MUST be the absolute path component (starting with `/`) to which
  the challenge binds.
- **`query`**: MUST be the raw query string without a leading `?`. The empty
  string MUST be used when no query is present.
- **`req_headers_sha256`**: MUST be the SHA-256 digest (hex) of the canonical
  request-header binding string defined below.
- **`req_body_sha256`**: MUST be the SHA-256 digest (hex) of the request body
  bytes. For empty bodies, the digest MUST be SHA-256 of the empty byte string.
- **`payee_locking_script_hex`**: MUST be the locking script that receives the
  payment output.
- **`nonce_utxo`**: MUST identify a specific UTXO that MUST be spent by the
  payment transaction. `nonce_utxo.satoshis` MUST be greater than zero.

Nonce issuance:

The `nonce_utxo` referenced in a challenge MUST originate
from the server issuing the challenge (or a service acting
on behalf of that server). Clients MUST NOT supply arbitrary
nonce UTXOs.

- **`expires_at`**: MUST be a UNIX timestamp in seconds. The server MUST reject
  proofs for challenges where the current time is strictly greater than
  `expires_at`.
- **`require_mempool_accept`**: If `true`, the server MUST verify mempool
  acceptance (Section 7) before executing the request.

Canonical header-binding string:

To compute `req_headers_sha256`, the server and client MUST construct a UTF-8
string by:

1. Selecting the request headers the server intends to bind (the selection
   policy is implementation-defined but MUST be consistent for a given server).
2. Lowercasing each selected header name.
3. Trimming optional whitespace around the header value.
4. Sorting headers by header name in ascending byte order.
5. Concatenating each as `name:value\n` (a single LF).

The SHA-256 digest of the resulting UTF-8 bytes MUST be encoded as lowercase
hex and placed into `req_headers_sha256`.

## 5. Payment Proof

### X402-Proof

The `X402-Proof` header MUST contain a base64url encoded JSON object
representing the payment proof.

Example:

```
GET /resource
X402-Proof: <base64url(JSON)>
```

The client MUST retry the original request with an `X402-Proof` header that
validly satisfies the server challenge.

The client MUST NOT reuse a proof for a different request binding.

### Proof Object

The proof object MUST contain the following fields.

```
{
  "v": 1,
  "scheme": "bsv-tx-v1",

  "challenge_sha256": "<hex>",

  "request": {
    "method": "GET",
    "path": "/v1/resource",
    "query": "",
    "req_headers_sha256": "<hex>",
    "req_body_sha256": "<hex>"
  },

  "payment": {
    "txid": "<hex>",
    "rawtx_b64": "<base64>"
  }
}
```

Rules:

• `challenge_sha256` MUST reference the original challenge  
• `txid` MUST match the transaction hash of `rawtx_b64`  

The server MUST reject proofs whose `v` value it does not support.

Additional requirements:

- **`v`**: MUST indicate the proof object version.
- **`scheme`**: MUST indicate the proof encoding scheme. If a server does not
  support the scheme, it MUST reject the proof (Section 9).
- **`challenge_sha256`**: MUST equal the value defined in the
  Challenge Hash subsection of this section, computed from the canonical
  challenge JSON bytes.
- **`request`**: MUST match the request-binding fields from the referenced
  challenge: `method`, `path`, `query`, `req_headers_sha256`, and
  `req_body_sha256`.
- **`payment.rawtx_b64`**: MUST be a base64 encoding of the raw transaction
  bytes.
- **`payment.txid`**: MUST equal the transaction identifier computed from the
  decoded raw transaction bytes.

### Challenge Hash

The value `challenge_sha256` MUST be computed as:

```
SHA256(canonical_challenge_json_bytes)
```

Where `canonical_challenge_json_bytes` is the canonical JSON serialization
of the decoded challenge object (after base64url decoding of the
`X402-Challenge` header) according to the Canonical JSON Encoding
rules defined in Section 2.

The resulting 32-byte digest MUST be encoded as lowercase hexadecimal.

## 6. Transaction Requirements

The payment transaction MUST satisfy the following conditions.

1. The transaction MUST include an input whose outpoint exactly matches
   the `nonce_utxo` (`txid`, `vout`) defined in the challenge.

2. The transaction MUST include an output whose locking script matches
   `payee_locking_script_hex`.

3. The payment output MUST contain a value greater than or equal to
   `amount_sats`.

4. The transaction MUST be syntactically valid and the `txid` carried in the
   proof MUST match the decoded transaction bytes.

5. The transaction MUST be valid under the settlement rules implied by
   `scheme`. If the server cannot validate the settlement rules for the
   challenge scheme, it MUST reject the proof (Section 9).

### Transaction Identifier

The `txid` value MUST be the double SHA-256 hash of the serialized
transaction bytes as defined by the settlement layer.

For Bitcoin-style transaction formats:

```
txid = SHA256(SHA256(raw_transaction_bytes))
```

The result MUST be encoded as lowercase hexadecimal.

## 7. Verification Procedure

Upon receiving a request containing `X402-Proof` the server MUST:

1. Decode the proof object.
2. Verify the challenge reference.
3. Validate request binding.
4. Check expiration conditions.
5. Decode the transaction.
6. Verify the nonce UTXO is spent.
7. Verify the payment output.
8. Optionally verify mempool acceptance.

If all conditions are satisfied the server MAY execute the request.

The server MUST enforce the following validation rules:

- **Decode**: If the proof is not valid base64url or not valid JSON, the server
  MUST reject it (Section 9).
- **Challenge reference**: The server MUST recompute `challenge_sha256` and MUST
  compare it to the provided value.
- **Request binding**: The server MUST ensure the proof `request` fields match
  both the referenced challenge and the actual inbound HTTP request.
- **Expiration**: The server MUST reject proofs for challenges where the
  current time is strictly greater than `expires_at`.
- **Nonce spend**: The server MUST verify the payment transaction spends the
  `nonce_utxo` outpoint (`txid`, `vout`) referenced in the challenge.
- **Payment output**: The server MUST verify an output pays at least
  `amount_sats` to `payee_locking_script_hex`.
- **Mempool acceptance**: If `require_mempool_accept` is `true`, the server
  MUST verify the payment transaction has been accepted to the mempool (or an
  equivalent transaction acceptance signal from the settlement layer) before
  executing the request.

### Request Binding Validation

The server MUST verify that the inbound HTTP request matches the
binding fields defined in the challenge and proof:

• HTTP method  
• request path  
• query string  
• header hash  
• body hash  

If any of these fields differ, the server MUST reject the proof.

## 8. Replay Protection

Replay protection MUST be achieved through the settlement
layer's double-spend protection mechanism.

The payment transaction MUST spend the `nonce_utxo`
identified in the challenge.

The server MUST verify that the transaction includes an
input whose outpoint exactly matches:

```
(txid, vout)
```

specified in `nonce_utxo`.

Because a UTXO can only be spent once, any subsequent
transaction attempting to reuse the same nonce UTXO
will constitute a double spend and MUST be rejected.

Servers MUST NOT rely on persistent nonce tracking for
replay protection correctness. Replay protection is
derived solely from settlement-layer double-spend
protection of the nonce UTXO.

## 9. Error Handling

The server MUST reject invalid proofs with an appropriate HTTP status.

Expired challenge → 402 Payment Required  
Nonce already spent → 402 Payment Required  
Invalid transaction → 400 Bad Request  
Request binding mismatch → 400 Bad Request  

Additional error handling requirements:

- If `X402-Proof` is missing or invalid, and payment is required, the server
  MUST respond with 402 and MUST include `X402-Challenge`.
- If the proof scheme is unsupported, the server MUST respond with
  400 Bad Request.
- If the payment amount is insufficient relative to `amount_sats`, the server
  MUST respond with 402 Payment Required.

## 10. Deployment Modes

x402 supports multiple deployment configurations. These configurations MUST
NOT change the protocol semantics, challenge format, proof format, or
verification procedure.

Client-funded  
The client constructs and funds the payment transaction.

Fee-delegated  
A third party MAY sponsor miner fees for the transaction.

Sponsored  
The server MAY sponsor the settlement transaction and/or miner
fees internally.

## 11. Implementation Status

An experimental reference implementation of the x402 protocol
is available.

Reference implementation repositories may include:

• x402-gateway  
• x402-client libraries  
• interoperability test harnesses  

These implementations are intended to validate the
protocol design and provide examples for implementers.

The protocol specification remains authoritative
over any particular implementation.

## 12. Security Considerations

Servers SHOULD consider:

• challenge expiration windows  
• mempool race conditions  
• rate limiting unpaid requests  
• transaction template validation

Clients SHOULD verify:

• challenge authenticity  
• payment destination  
• request binding integrity
