## Abstract

x402 is a stateless settlement-gated HTTP protocol that allows
servers to require verifiable economic settlement before executing
a request.

This document defines the canonical protocol specification.

## 1. Introduction

HTTP status code **402 Payment Required** has existed since the early
days of the web but was never formally standardized.

x402 defines a deterministic protocol for using HTTP 402 to require
verifiable economic settlement before executing a request.

In x402, payment functions as an authorization primitive.

Instead of relying on accounts, API keys, or subscriptions,
the client proves payment directly as part of the request flow.

---

## 2. Protocol Overview

The x402 protocol follows a simple request-response pattern.

1. Client requests a protected resource  
2. Server responds with HTTP 402 and a challenge  
3. Client performs settlement  
4. Client retries request with payment proof  
5. Server verifies proof and serves the resource  

```
Client → Request resource
Server → 402 Payment Required (challenge)
Client → Retry request with proof
Server → 200 OK
```

The server does not maintain client accounts.
Authorization is derived solely from proof of settlement.

---

## 3. HTTP Challenge

When payment is required, the server responds with:

```
HTTP/1.1 402 Payment Required
X402-Challenge: <base64url(JSON)>
```

The challenge defines the settlement conditions required for
the request to be executed.

---

## 4. Payment Proof

The client retries the request with a payment proof.

```
GET /resource
X402-Proof: <base64url(JSON)>
```

The proof contains a settlement transaction that satisfies
the challenge requirements.

---

## 5. Verification Rules

Upon receiving a request containing `X402-Proof`,
the server verifies:

1. proof integrity  
2. challenge binding  
3. request binding  
4. expiry conditions  
5. payment transaction validity  

If all checks succeed, the server executes the request.

---

## 6. Replay Protection

x402 prevents replay using the settlement layer’s
native double-spend protection.

Each challenge includes a unique nonce UTXO that must
be spent in the payment transaction.

Because a UTXO can only be spent once, the proof is
single-use.

---

## 7. Error Conditions

Servers may reject proofs for several reasons:

• challenge expired  
• nonce already spent  
• insufficient payment  
• request mismatch  
• invalid transaction  

In these cases the server returns HTTP 402 or HTTP 400.

---

## 8. Deployment Modes

x402 supports multiple deployment configurations.

Client-funded  
Client constructs and funds the payment transaction.

Fee-delegated  
A delegator service sponsors miner fees for the transaction.

Sponsored  
The server sponsors settlement internally.

These configurations do not change the protocol semantics.

---

## 9. Security Considerations

Servers should consider:

• challenge expiration windows  
• mempool race conditions  
• rate limiting unpaid requests  
• transaction template validation  

Clients should verify:

• challenge authenticity  
• payment destination  
• request binding integrity  

---

## Summary

x402 transforms HTTP 402 into a deterministic economic protocol.

Requests are executed only after verifiable settlement,
allowing stateless pay-per-request services without
accounts or billing infrastructure.
